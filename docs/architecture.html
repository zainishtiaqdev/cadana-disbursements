Disbursements — Architecture & Design

A payroll-ops dashboard: review pending payouts, run a batch, and watch per-worker results land live — built to stay correct when the payment provider flakes. Go backend, Vue 3 + TS frontend, optional Postgres for durable state.

1 · System overview

Three tiers. The browser holds only UI state and talks to one Go service over a tiny JSON API. The service owns all business logic and writes through a Store interface that has two adapters: an in-memory map (the zero-setup default) and Postgres (durable, for the live demo). The payment rail is an interface too, so it can be a flaky mock in dev and a deterministic stub in tests.

Why a single long-running Go process (not serverless): idempotency and batch state live behind the Store. With the in-memory adapter that state is in-process, so the API must be one instance. With Postgres it is durable and could scale horizontally — the same interface, no logic change.

2 · Request lifecycle

POST accepts the work and returns immediately; payment happens in the background so the provider's latency never blocks the request. The UI then polls until every row has settled — which is exactly what makes failures and timing visible to the operator. 

sequenceDiagram
  autonumber
  participant U as Vue SPA
  participant A as Go API
  participant S as Service
  participant P as Provider
  U->>A: POST /disbursements {batch_id, worker_ids}
  A->>S: Submit(batch_id, ids)
  S->>S: CreateBatch — create-if-absent
  alt new batch_id
    S-->>A: batch (all pending)
    A-->>U: 202 Accepted
    par background, bounded fan-out
      S->>P: Pay(worker A)
      S->>P: Pay(worker B)
      S->>P: Pay(worker …)
    end
    P-->>S: txn id / error per worker
  else existing batch_id
    S-->>A: existing batch (provider NOT called)
    A-->>U: 202 Accepted
  end
  loop poll until no row is pending
    U->>A: GET /disbursements/:batch_id
    A-->>U: current per-worker status
  end

3 · Idempotency — the crux

batch_id is supplied by the client and used as an idempotency key (the same pattern as a Stripe idempotency key). Creating a batch is an atomic create-if-absent; a resubmission of the same id returns the existing batch and never touches the provider — so a retry, a double-click, or a network replay can't double-pay. 

flowchart TD
  R["POST /disbursements (batch_id)"] --> C{"batch_id already exists?"}
  C -->|"no — first time"| N["insert batch + N pending rows
spawn async payment"] --> OK["202 · all pending"]
  C -->|"yes — replay"| E["return existing batch
provider is NOT called"] --> OK2["202 · current state"]

In-memory

A single RWMutex guards the batch map; CreateBatch checks-and-inserts under the lock, so concurrent identical submits resolve to exactly one creator.

Postgres

INSERT INTO batches(id) … ON CONFLICT DO NOTHING. The primary key is the idempotency guarantee — it holds across restarts and across multiple instances, not just within one process.

Retrying failed workers re-batches them under a new id. Already-paid workers are never in that set, so there is nothing to double-pay.

4 · Concurrency model

A batch fans out to one goroutine per worker, capped by a semaphore so we never stampede the provider, and joined with a WaitGroup. Each result is written under the store lock. One worker failing is just one failed row — it never blocks or rolls back the others. The whole suite runs under go test -race.

  PAY -->|"ok"| US["UpdateResult: success + txn id"]
  PAY -->|"error"| UF["UpdateResult: failed + reason"]

  UF --> WG

6 · Frontend reactivity

State lives in composables, not components. Raw server data is held in refs; everything derived is computed, so the summary and the failed-set can never drift out of sync with the rows.

PieceKindWhy
resultsrefraw server truth, replaced on each poll
summarycomputedcounts derived from results — single source
failedWorkerIdscomputeddrives the "Retry failed" action
useWorkerscomposableroster fetch + loading/error, reused independently
useDisbursementBatchcomposableowns submit → poll → cleanup; onScopeDispose clears the timer

The API contract is one hand-written module (types/api.ts) mirroring the Go DTOs — a single source of truth, chosen over codegen because the surface is tiny and stable.

8 · Decisions & trade-offs

DecisionWhy
Client-supplied batch_id as idempotency keyMakes "don't double-pay on resubmit" provable and testable; mirrors real payment APIs.core
Async processing + pollingSurfaces provider latency & failures to the operator instead of hiding them in one slow request. Brief allows polling.core
shopspring/decimal, string on the wireNo float rounding on money, across multiple currencies.core
Store behind an interface; in-memory defaultOne-command local run (honors the brief) and a durable path for the demo — no logic changes between them.core
Polling, not SSE/WebSocket pushTime-box trade-off: simpler, fewer moving parts, adequate at this scale. Push would cut redundant requests.trade-off
Hand-written shared types (no codegen)Tiny, stable surface — one type module is simpler than an OpenAPI pipeline here.trade-off
